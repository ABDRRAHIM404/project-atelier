# Version 1 Database Design

**Project:** Project Atelier / بيتي بذوقي  
**Status:** Accepted by the Product Owner on 2026-07-16  
**Phase:** Database and API design  
**Database:** PostgreSQL on the approved Supabase managed service  
**Data access:** Approved Drizzle server data layer with reviewed SQL migrations  
**Important:** This document is a physical/logical design specification. It contains no DDL, migration, Supabase provisioning, or application code.

## 1. Design authority

This design implements the approved contracts in `MASTER_PRD.md`, `DOMAIN_MODEL.md`, `STATE_MACHINES.md`, `MODULE_BOUNDARIES.md`, `DATA_ARCHITECTURE.md`, `SECURITY_ARCHITECTURE.md`, `AUTH_ARCHITECTURE.md`, `STORAGE_ARCHITECTURE.md`, and `ADR_INDEX.md`.

The following rules dominate every table-level choice:

1. Customer acceptance of the current sent Quotation Revision creates exactly one Order.
2. Sent Quotation Revisions, accepted revisions, Customer Acceptance, Order Item Snapshots, Payment Submissions, Payment Verifications, Production Updates, and Audit Events are append-only or immutable at their documented boundary.
3. Production state belongs to the Order in Version 1 and cannot leave `NOT_STARTED` without verified payment.
4. Customers access only their own private data and files.
5. File bytes live in approved AWS S3 zones; PostgreSQL stores authoritative ownership, classification, lifecycle, and exact object-version references.
6. Arabic is required. English is optional and publication-gated. French has no Version 1 application data path.
7. Version 1 is one business and one Manager. No tenant discriminator is copied into every row.
8. Unresolved Business Policy and Configuration values are not given defaults in this design.

## 2. PostgreSQL organization

One PostgreSQL database is divided into module-owned schemas. Schemas are ownership and review boundaries, not microservices.

| Schema | Owning module | Main responsibility |
|---|---|---|
| `iam` | Access and Identity | Local principals, Customers, Manager, external identity links |
| `config` | Business Configuration | Singleton business identity, locations, typed configuration revisions |
| `catalog` | Catalog and Search | Products, catalog structures, allowed configurations, public search projection |
| `cms` | CMS and Localization | CMS versions, localizable resources, translation revisions and publication |
| `projects` | Customer Projects | Draft Projects, items/configurations, immutable submitted snapshots |
| `quotes` | Quotations and Acceptance | Quotations, immutable revisions/items, responses, acceptance |
| `orders` | Orders | Order aggregate and immutable accepted snapshots |
| `payments` | Payments | Proof submissions, Manager verification decisions, current payment summary |
| `production` | Production | Order-level current production state and append-only updates |
| `fulfilment` | Fulfilment | Accepted fulfilment terms, current state, handoff evidence |
| `messaging` | Messaging | One continuous Customer conversation, immutable Messages, read positions |
| `notifications` | Notifications | In-app notifications, channel deliveries, attempts, template references |
| `files` | Files and Media | Upload intents, object metadata/version, scan events, attachment relations |
| `audit` | Audit and Operations | Append-only business/security Audit Events |
| `ops` | Audit and Operations | Idempotency, outbox, jobs, attempts, inbound provider events |

The migration owner owns database objects. Runtime and job roles are non-owner roles with explicit grants and no `BYPASSRLS`. Browser clients receive no direct table credentials and do not use Supabase Data APIs as the business API.

## 3. Common data conventions

### 3.1 Identifiers and versions

- Internal and externally exposed business identifiers use PostgreSQL `uuid` values generated by the trusted server/database path.
- Human-readable Order or quotation references are separate unique display references and never authorization credentials.
- Mutable aggregate roots contain an integer `record_version` used for optimistic concurrency.
- Append-only sequences use a parent-scoped integer sequence with a uniqueness constraint, such as quotation revision number or Production Update sequence.
- External provider identifiers are stored in provider-specific reference columns/tables and never replace local identifiers.

### 3.2 Time and actor attribution

- Instants use timezone-aware timestamps and are stored as UTC.
- Every mutable record has `created_at` and `updated_at`; immutable records have `created_at` plus event-specific occurrence fields.
- Manager/Customer actions reference the local `iam.principals` identity, not an email address or provider subject.
- Business-local display and deadline calculation use the configured IANA time zone; this design does not assume its value.

### 3.3 Money, units, and locale

- Final monetary values use signed 64-bit integer minor units plus an ISO 4217 currency code.
- Currency exponent/reference metadata is retained so configurable currencies do not require floating-point arithmetic.
- SAR is the initial configured default, but each commercial snapshot stores its own currency.
- Dimensions use exact numeric values with an explicit unit code. Original Customer text is preserved separately from normalized numeric values where both are needed.
- Locale codes reference an enabled locale registry. Arabic is mandatory; English can be enabled by approved configuration. French remains disabled in Version 1.

### 3.4 Deletion and foreign keys

- Historical/commercial foreign keys use `RESTRICT` or equivalent no-delete behavior.
- No cascade may remove a sent quotation, acceptance, Order snapshot, Payment record, Production Update, fulfilment proof, Message, Notification evidence, or Audit Event.
- Hard deletion APIs are absent until the relevant retention/deletion policy is approved.
- Draft-only dependent data may eventually use controlled deletion, but this phase does not authorize a general delete policy.

### 3.5 JSON use

JSONB is permitted only for bounded, versioned data whose shape varies by approved resource type: configuration selections, immutable display snapshots, CMS block structures, safe provider diagnostics, and typed configuration values. Every such column records a schema version and is validated at the application boundary. JSONB does not replace relational ownership, lifecycle, money, actor, or state fields.

## 4. Identity and business configuration

### 4.1 `iam.principals`

Stable local authorization identity.

| Field group | Design |
|---|---|
| Identity | `id`, `actor_type` (`CUSTOMER` or `MANAGER`) |
| Access | `access_status` limited to security availability (`ACTIVE` or `DISABLED`), `disabled_at`, safe reason reference |
| History | creation/update timestamps, `record_version` |

Account closure, anonymization, export, and legal deletion are not encoded as extra lifecycle states until `BP-007` is approved.

### 4.2 `iam.external_identities`

Maps a provider subject to one local principal.

Key facts: provider identifier, provider subject, principal, link status, verified email snapshot at link time, linked/unlinked timestamps, and attribution for recovery changes. The pair `(provider, provider_subject)` is unique. Only one active primary identity link is allowed per principal for Version 1. Provider metadata cannot set `actor_type`.

### 4.3 `iam.customers`

One-to-one Customer profile for a Customer principal. Contains stable Customer ID, principal ID, verified/contact email snapshots as allowed by the provider flow, preferred locale, approved profile fields, timestamps, and `record_version`. Fields beyond verified email and locale are added or enabled only through `CFG-006`; no unsupported required profile field is invented.

### 4.4 `iam.managers`

One-to-one Manager profile for a Manager principal. A singleton marker and partial uniqueness constraint permit at most one active Version 1 Manager. The record contains no password, TOTP secret, or recovery code. Bootstrap/replacement attribution is auditable.

### 4.5 `config.business_profile`

A singleton business boundary containing stable business identity, operating country, default currency reference, default locale, configured time zone, and approved contact fields. It is not a tenant table and is not referenced as a tenant key by every business row.

### 4.6 `config.fulfilment_locations`

Represents the home workshop and future physical showroom/pickup locations without introducing tenancy. Fields include stable identity, location kind, status, localizable name reference, structured address, and public-contact fields. Only approved active locations may be used in a new quotation. Accepted Orders snapshot location/address facts.

### 4.7 Typed configuration revisions

`config.configuration_definitions` contains a code-owned allowlist of configuration keys, owner module, value kind, schema version, scope, sensitivity, and whether Manager or Operator approval is required. Manager users cannot create executable configuration keys.

`config.configuration_revisions` contains definition key, typed/versioned value, lifecycle (`DRAFT`, `ACTIVE`, `RETIRED`), effective interval, actor/approver, reason, and timestamps. At most one revision is active for a singleton key at a given instant. Activation is audited and previous revisions remain immutable.

The initial registry can represent `CFG-001` through `CFG-008`, but no unanswered duration, threshold, file limit, retention period, rounding rule, service area, or notification preference receives a guessed value. Security invariants and historical immutability are not configuration keys.

## 5. Catalog, configuration, and localization

### 5.1 Localized resource pattern

`cms.localized_resources` is a stable parent for translatable domain objects. Catalog and CMS records reference one unique resource ID. `cms.translation_revisions` stores:

- resource and locale;
- parent-scoped revision number;
- lifecycle (`DRAFT`, `IN_REVIEW`, `APPROVED`, `PUBLISHED`);
- schema version and typed content JSON;
- Arabic source revision reference for optional English;
- author, human reviewer/approver, timestamps, and content digest;
- prior revision link and stale-source marker.

There is at most one currently published revision per resource and locale. `APPROVED` and `PUBLISHED` rows are immutable. A changed Arabic source can mark an English revision stale but cannot rewrite it. Publication requires human Manager attribution.

### 5.2 Catalog tables

| Relation | Key fields and constraints |
|---|---|
| `catalog.categories` | ID, localized resource, status, sort order, version; non-destructive retirement |
| `catalog.collections` | ID, localized resource, status, sort order, media relationship, version |
| `catalog.products` | ID, localized resource, category, furniture type, lifecycle state, starting money, production-information fields, version |
| `catalog.product_collections` | Product/Collection pair; unique membership |
| `catalog.materials` | ID, localized resource, availability/status, version; no inventory quantity |
| `catalog.colors` | ID, localized resource, safe display value, availability/status, version |
| `catalog.product_materials` | Product/Material assignment and availability/order |
| `catalog.product_colors` | Product/Color assignment and availability/order |
| `catalog.product_options` | Product, localized resource, bounded option kind, required flag, order, lifecycle, version |
| `catalog.product_option_values` | Option, localized resource, stable machine value, availability/order, version |
| `catalog.product_option_exclusions` | Same-product option-value pair that cannot be selected together; canonical ordering prevents duplicate pairs |
| `catalog.product_option_dependencies` | Bounded requirement/allowance between two option values; no formulas or executable expressions |
| `catalog.product_dimension_rules` | Product, dimension kind, rule kind (`FIXED`, `RANGE`, `FREE`), unit, exact bounds/fixed value; consistency checks by rule kind |

Product lifecycle values are `DRAFT`, `PUBLISHED`, `HIDDEN`, `TEMPORARILY_UNAVAILABLE`, and `ARCHIVED`. Only `PUBLISHED` is publicly visible and orderable. Archive is non-destructive.

### 5.3 Catalog search projection

`catalog.search_documents` is a rebuildable read projection keyed by Product and locale. It stores only published Customer-safe text, source revision identifiers, normalized Arabic search text, PostgreSQL search vector, and freshness timestamp. It never contains private messages, Customer records, payment material, or restricted notes. Consumers cannot write through this projection.

### 5.4 CMS tables

`cms.contents` is a stable page/section/content identity with kind, stable slug/reference, current visibility, and current published-version pointer.

`cms.content_versions` contains content ID, revision number, versioned block structure JSON, localized resource ID, draft/publication state, creator, publisher, timestamps, and digest. A published version is immutable; edits create a new draft version. Hiding changes the stable content visibility while retaining the published version.

No generic block type is executable. Each CMS content kind and block schema is code-owned and versioned. Policy/legal content publication remains subject to `BP-010`.

## 6. Customer Projects and submitted history

### 6.1 Draft relations

| Relation | Key fields and constraints |
|---|---|
| `projects.customer_projects` | ID, owning Customer, state (`DRAFT`, `SUBMITTED`), name/notes, submission timestamp, version |
| `projects.project_items` | ID, Project, Product, position, Customer notes, configuration reference, timestamps, version; one row represents one physical requested item because quantity is not approved |
| `projects.product_configurations` | ID, Project Item unique, Product, configuration schema version, bounded selections JSON, validation/catalog reference, last validated timestamp, version |

Only the owning Customer can mutate a `DRAFT` Project. Items and configurations become non-editable when the Project becomes `SUBMITTED`.

### 6.2 Submission snapshot relations

| Relation | Key fields and constraints |
|---|---|
| `projects.submitted_requests` | ID, source Project unique, Customer, state, submitted timestamp, current workflow version |
| `projects.submitted_request_items` | ID, Request, source Project Item, sequence, Product reference, immutable Product/configuration/display snapshot JSON and schema version |
| `projects.request_clarifications` | Request, Message reference, direction/purpose, occurrence timestamp; it links append-only communication without changing submitted content |

Submitted Request item rows are immutable. The Request state may change only through approved transitions. There are no Version 1 columns or transitions for withdrawal, expiry, infeasibility closure, or reopening until `BP-001`/`CFG-001` approves them.

## 7. Quotations and acceptance

### 7.1 Relations

| Relation | Key fields and constraints |
|---|---|
| `quotes.quotations` | ID, Submitted Request unique, Customer, lifecycle, current sent revision pointer, next revision number, version |
| `quotes.quotation_revisions` | ID, Quotation, revision number, state, currency, production estimate snapshot, fulfilment method/details, delivery address snapshot, totals, terms/notes, author, sent timestamp, digest |
| `quotes.quotation_items` | ID, Revision, source Submitted Request Item, sequence, immutable item/configuration/display snapshot, item total and currency |
| `quotes.price_components` | ID, Revision, optional Quotation Item, ordered component kind/label snapshot, signed minor-unit amount, currency, tax metadata where approved |
| `quotes.quotation_responses` | ID, Revision, Customer, outcome (`CHANGES_REQUESTED` or `DECLINED`), Message/reason reference, timestamp; immutable |
| `quotes.customer_acceptances` | ID, Revision unique, Quotation unique, Customer, accepted digest/terms reference, actor/session evidence allowed by privacy policy, accepted timestamp; immutable |

### 7.2 Required constraints

- `(quotation_id, revision_number)` is unique.
- At most one current `SENT` revision exists per Quotation.
- Only a `DRAFT` revision and its child items/components may be updated.
- Once sent, revision, items, components, delivery/address snapshot, terms, totals, and digest reject update/delete.
- `ACCEPTED` is terminal and can never be superseded.
- The Customer Acceptance revision must be the current sent revision belonging to that Customer.
- Quotation component totals and stored revision totals are validated atomically by a deferred consistency rule/application transaction.
- Expiry has no automatic database behavior until the policy and configured value are approved.

## 8. Orders and immutable commercial snapshots

### 8.1 Relations

| Relation | Key fields and constraints |
|---|---|
| `orders.orders` | ID, display reference, Customer, Acceptance unique, accepted Revision unique, lifecycle state, currency, accepted total, creation/completion timestamps, version |
| `orders.order_item_snapshots` | ID, Order, source Quotation Item unique, sequence, Product reference, immutable item/configuration/display snapshot, item total/currency |
| `orders.order_price_components` | ID, Order, optional Order Item Snapshot, sequence, immutable component kind/label/tax snapshot, signed amount/currency |
| `orders.order_terms_snapshots` | Order unique, accepted terms/schema version, production estimate, Customer/contact facts needed for history, digest |

The acceptance transaction copies all facts needed to interpret the commercial agreement. Historical rendering must not require current Product, Translation, Material, Color, option, address, configuration, or configuration-policy values.

Order Item Snapshots contain no production-state column, no production version, and no Production Update relation. A future item-level production feature may add a new table that references the immutable Order Item Snapshot; it cannot alter Version 1 history.

### 8.2 Order lifecycle constraints

Order states are `AWAITING_PAYMENT`, `PAYMENT_UNDER_REVIEW`, `PAYMENT_VERIFIED`, `IN_PRODUCTION`, `READY_FOR_FULFILMENT`, `COMPLETED`, plus reserved `CANCELLED` with no enabled transition. Current-state changes require approved transition pairs and optimistic/locked concurrency checks.

Exactly one Order may be created for one Customer Acceptance/accepted Revision. Creation of Acceptance, Order, items, price components, terms, initial Payment summary, initial Production state, Fulfilment, Audit Event, and outbox intents is atomic.

## 9. Payment design

| Relation | Key fields and constraints |
|---|---|
| `payments.order_payment_status` | Order primary key, current state, current submission, verified decision, version |
| `payments.payment_submissions` | ID, Order, Customer, proof Attachment unique, security-processing state, submitted/eligible-for-review timestamps, immutable declared metadata |
| `payments.payment_verifications` | ID, Order, Submission, Manager, outcome (`VERIFIED` or `REJECTED`), safe reason/note, decision timestamp; immutable |

Every accepted proof submission creates a new immutable Payment Submission; no replacement overwrites a prior row or object. Upload Intent, File Object, and scan-event history preserve failed or rejected upload processing separately. A file that is not `CLEAN` cannot create a reviewable Payment Submission or move the Order payment state to `SUBMITTED`.

Required uniqueness includes one Manager outcome per submission under the current policy and at most one `VERIFIED` Payment Verification per Order. A rejected submission can be followed only by a new submission. `VERIFIED` cannot be rewritten or reversed until `BP-003` defines an append-only correction path.

Payment amount exception fields/reasons may be represented by approved configuration later, but no partial, excessive, duplicate, fraudulent, or automatic matching outcome is enabled now.

## 10. Order-level production and fulfilment

### 10.1 Production

| Relation | Key fields and constraints |
|---|---|
| `production.order_production` | Order primary key, current state, next sequence, start/ready timestamps, version |
| `production.production_updates` | ID, Order, sequence, from/to state, Manager, optional Customer-visible note, safe reason reference, occurred timestamp; immutable |

Allowed pairs are exactly:

- `NOT_STARTED` → `MATERIALS_PREPARATION`
- `MATERIALS_PREPARATION` → `IN_PRODUCTION`
- `IN_PRODUCTION` → `QUALITY_INSPECTION`
- `QUALITY_INSPECTION` → `IN_PRODUCTION`
- `QUALITY_INSPECTION` → `READY`

The first transition requires an authoritative verified Payment Verification in the same transaction. `(order_id, sequence)` is unique. No skipped state or item-level production record is permitted.

### 10.2 Fulfilment

| Relation | Key fields and constraints |
|---|---|
| `fulfilment.fulfilments` | ID, Order unique, accepted method, accepted location/address/price snapshot, state, ready/handoff timestamps, version |
| `fulfilment.handoff_proofs` | ID, Fulfilment, clean Attachment, captured-by principal, configured evidence metadata/schema version, captured timestamp; immutable |

Delivery requires an accepted address snapshot and quoted delivery price; pickup references an accepted active fulfilment location snapshot. The method and accepted facts are immutable after Order creation. Completion requires at least one authorized clean handoff proof.

No tables or states for dispatch, failed attempt, refusal, damage, partial receipt, dispute, compensation, or rescheduling are activated until `BP-006`/`CFG-005` is approved. A future additive attempt table can be introduced without changing completed handoff facts.

## 11. Messaging and notifications

### 11.1 Messaging

| Relation | Key fields and constraints |
|---|---|
| `messaging.conversations` | ID, Customer unique, created timestamp; one continuous Version 1 conversation |
| `messaging.messages` | ID, Conversation, sender principal, body/plain-text or approved safe content, optional Project or Order context, client message key, sent timestamp; immutable content |
| `messaging.read_positions` | Conversation/principal unique, last read Message, read timestamp, version |
| `messaging.message_attachments` | Message/Attachment pair, ordering; exact parent authorization |

A Message may reference at most one Customer Project or Order context, and the referenced object must belong to the conversation Customer. Message edits/deletion are not enabled while retention policy remains open.

### 11.2 Notifications

| Relation | Key fields and constraints |
|---|---|
| `notifications.notifications` | ID, recipient principal, originating event ID, event type, resource reference, locale, template key/version, created/read timestamp |
| `notifications.channel_deliveries` | ID, Notification, channel (`IN_APP`, `EMAIL`), status, provider reference, next attempt/terminal fields, version |
| `notifications.delivery_attempts` | ID, Delivery, attempt number, started/completed time, outcome, safe error code, provider receipt; append-only |

Notification creation is unique by recipient, originating business event, and channel. Read state never grants access to the linked resource. The seven accepted event types are implemented; optional events/preferences remain disabled/configuration-gated.

## 12. File and media metadata

### 12.1 Relations

| Relation | Key fields and constraints |
|---|---|
| `files.upload_intents` | ID, requesting principal, purpose, target reference, declared filename/type/size, expected checksum where available, generated object key, expiry, state, idempotency key |
| `files.file_objects` | ID, uploader, classification, purpose, logical zone, immutable S3 key/version, byte size, declared/detected type, checksum, lifecycle/scan state, timestamps |
| `files.scan_events` | ID, File Object, provider event reference, outcome, engine/provider metadata, occurred/received time, safe reason; append-only |
| `files.attachments` | ID, File Object unique, uploader/owner Customer where applicable, attachment kind, availability state, creation timestamp |
| Parent link tables | `request_attachments`, `payment_proof_attachments`, `fulfilment_proof_attachments`, `catalog_media`, and `cms_media`, each with real foreign keys and parent-specific uniqueness |

Object key, bucket/zone, and version are server-owned. A presigned URL is never stored. The database distinguishes `PENDING_UPLOAD`, `UPLOADED`, `SCAN_PENDING`, `CLEAN`, `QUARANTINED`, and `REJECTED`; deletion remains a governed future action. Unknown/failed scan is not clean.

Payment proof Attachment rows accept only detected JPG, PNG, or PDF. Exact byte/page/dimension limits reference `CFG-003` and must be approved before external upload release.

## 13. Audit, idempotency, outbox, jobs, and webhooks

### 13.1 Audit

`audit.events` stores immutable event ID/type, occurred time, actor kind/principal, target type/ID, operation, outcome, before/after state names where appropriate, safe reason, request/correlation ID, and allowlisted metadata/schema version. It excludes secrets, raw file content, tokens, presigned URLs, complete payment evidence, and uncontrolled request bodies.

Normal runtime roles have insert/select-by-authorized-contract only and no update/delete. Failed privileged attempts are written in an independent safe transaction when the database is available.

### 13.2 Operational durability

| Relation | Purpose |
|---|---|
| `ops.idempotency_records` | Actor/operation/target-scoped client request key, request digest, progress/outcome, minimal replay result and resource references |
| `ops.outbox_events` | Business event and side-effect intent committed with authoritative state; payload schema, correlation, dedupe key, availability and completion state |
| `ops.jobs` | Leased executable work derived from an outbox event or approved scheduled task; handler type/version, lease, attempt count, next eligibility, terminal state |
| `ops.job_attempts` | Append-only attempt timing/outcome/safe error details |
| `ops.inbound_provider_events` | Provider/event ID, payload digest, signature verification result, receive/process state, correlation, timestamps |

`(provider, provider_event_id)` is unique where a provider supplies a stable ID. Business uniqueness still protects against semantic duplicates with different provider IDs. Outbox and job payloads contain minimum safe facts, not complete domain objects.

## 14. Database-enforced invariants

The future DDL/migration implementation must prove the following with constraints, partial unique indexes, restricted grants, and narrowly scoped transition/immutability triggers where declarative constraints are insufficient:

1. One active Version 1 Manager.
2. One Customer owner for every private Customer aggregate.
3. One Submitted Request per submitted Project version.
4. One Quotation per Submitted Request and unique sequential revision numbers.
5. At most one current sent Quotation Revision per Quotation.
6. No update/delete of sent or accepted revision content and child line items.
7. One Acceptance and one Order per accepted revision.
8. No update/delete of Acceptance or Order Item Snapshots.
9. At most one verified Payment decision per Order; Payment history is append-only.
10. Production start requires verified payment and only approved state pairs.
11. Production Updates reference Order only, never Order Item Snapshot.
12. Fulfilment completion requires a clean handoff proof and method-consistent state.
13. Private-resource ownership is consistent across parent/child rows.
14. Published translation/CMS versions and Audit Events are immutable.
15. Outbox/idempotency/provider-event uniqueness prevents duplicate authoritative effects.

Application validation remains mandatory; database constraints are the final consistency boundary, not a replacement for useful errors.

## 15. Index and query plan

Indexes are justified by named access patterns and verified with representative data. Initial design candidates are:

- all foreign-key columns used in ownership/parent queries;
- Customer plus state/update time for Projects, Requests, Orders, Messages, and Notifications;
- state plus creation/next-eligible time for Manager queues, outbox, and jobs;
- Quotation plus revision number/current-state uniqueness;
- Order plus sequence/time for Payment, Production, Fulfilment, and timeline reads;
- recipient plus unread/created time for Notifications;
- conversation plus Message time/ID for cursor pagination;
- file state/scan state and object key/version uniqueness;
- audit target/time, actor/time, event type/time, and correlation ID;
- PostgreSQL GIN/GiST/trigram indexes only on the published catalog search projection.

Private list queries must use owner predicates in the indexed path. Offset pagination is not used for growing Messages, Notifications, Audit Events, or timelines.

## 16. Row-level security and database roles

Application authorization is the primary policy and is required at every server entry point. PostgreSQL row-level security is additional defense for Customer-owned tables, not a browser access strategy.

- Migration owner: owns objects; never used by the application runtime.
- Runtime role: non-owner, no bypass, explicit grants, RLS forced on Customer-owned relations.
- Job role: limited to claimed job/outbox operations and the application command it executes; no unrestricted Manager identity.
- Read-only operations role: approved diagnostics only; sensitive/restricted access remains separate.
- Backup/recovery role: no normal application use.

Protected database work runs in a transaction that sets trusted, transaction-local actor/principal/Customer context before queries. RLS checks owner or approved Manager context. Module/application authorization still checks action and lifecycle because RLS alone cannot express the complete state machine safely. Connection reuse must never carry actor context outside its transaction.

## 17. Recovery and lifecycle

- Supabase PITR and independently encrypted logical exports protect relational history under the accepted RPO/RTO.
- S3 object versioning and database/object reconciliation protect file history.
- Every monthly restore exercise verifies identities, ownership, immutable constraints, outbox/jobs, Audit Events, and private-object references.
- No retention or purge job is enabled without approved class-specific policy, legal review where required, a dry-run/report mode, and restore evidence.
- Provider deletion never becomes the only way to represent account deactivation or content retirement.

## 18. Deferred relations not created for Version 1

Version 1 does not create active persistence for Favorites, Saved Designs, Reviews, item-level production, advanced analytics events/warehouse, AI prompts/results, push subscriptions, worker accounts, inventory, suppliers, multi-tenancy, refunds, returns, warranty/repair cases, disputes, delivery-attempt workflows, or advanced 360° processing.

Names in the conceptual domain remain roadmap context; they do not authorize dormant tables or APIs. New scope requires its policy decision and database/API review.

## 19. Approval record and implementation gates

The Product Owner approved this design on 2026-07-16, including:

- table and module ownership;
- snapshot completeness for Submitted Request, Quotation, Acceptance, Order, fulfilment, and localized history;
- database-enforced production/payment and immutability defenses;
- RLS/runtime-role model and Customer isolation test strategy;
- file metadata and scan gating;
- idempotency/outbox/job persistence;
- unresolved policy/configuration gates listed in the phase review report.

Implementation must still satisfy these obligations with executable migration, test, security, and review evidence. No database object has been created by this document or by the implementation-planning phase.
