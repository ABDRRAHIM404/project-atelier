# Version 1 Idempotency and Concurrency Rules

**Status:** Accepted by the Product Owner on 2026-07-16  
**Applies to:** API commands, database transactions, durable work, file finalization, and provider callbacks

## 1. Guarantee

Version 1 provides at-least-once request/delivery tolerance with exactly-once authoritative business outcomes where required by unique business facts. It does not claim exactly-once network delivery.

A retry may repeat transport and handler execution, but it must not create a second Order, second Customer Acceptance, duplicate Payment decision, skipped Production transition, duplicate handoff completion, duplicate Message, or duplicate Customer notification for the same event.

## 2. Defense layers

1. **Client key:** identifies a logical mutation attempt across timeout/retry.
2. **Request digest:** prevents one key being reused for different semantics.
3. **Database uniqueness:** protects the underlying business fact even if client/provider keys differ.
4. **State and version guard:** prevents stale or invalid transitions.
5. **Transactional idempotency record:** records progress and minimal replay outcome with the business transaction.
6. **Outbox/job deduplication:** prevents repeated side-effect intent/handler result.
7. **Provider key/event deduplication:** uses provider support but is never the only protection.

## 3. Client idempotency contract

Mutation endpoints marked “required” accept an `Idempotency-Key` header. The key is opaque, non-secret, generated by the client for one user intention, and reused only when retrying that exact command.

The server scopes a key by:

- authenticated local principal or trusted system/provider identity;
- API version and operation name;
- target aggregate/resource where applicable;
- supplied key.

The server calculates a canonical request digest from business-significant fields, normalized path parameters, and approved content references. Authentication tokens, correlation IDs, locale wording, timestamps not part of the command, and transport ordering do not create different semantics.

### Same key, same digest

- If still processing: return `IDEMPOTENCY_IN_PROGRESS` and safe retry guidance.
- If committed: replay the same status and minimal response/resource references after current authentication and ownership checks.
- If a terminal expected rejection was intentionally recorded: replay that rejection while its record remains available.

### Same key, different digest

Return `IDEMPOTENCY_KEY_REUSED`. Never run the new request.

### Different key, same business fact

Database/state uniqueness still returns the existing committed result where safe or a typed conflict. A different key cannot create a second accepted Order or verified payment.

## 4. Required operation matrix

| Operation | Client key | Business uniqueness / replay rule |
|---|---:|---|
| Customer first-login provisioning | Internal/provider key | One active local mapping per provider subject |
| Create Customer Project | Required | One Project ID returned for command key |
| Add Project Item | Required | One item for key; later explicit edit is a different command |
| Submit Customer Project | Required | One Submitted Request per Project version |
| Start Manager review | Recommended | State/version makes repeat a no-op/conflict |
| Request/provide clarification | Required when creating Message | Message client key unique within Conversation/sender |
| Create Quotation draft/revision | Required | Revision identity and parent-scoped number unique |
| Send Quotation Revision | Required | One send transition for Revision; current sent uniqueness |
| Request changes / decline | Required | One terminal/changes response per actionable revision |
| Accept Quotation Revision | **Required** | One Acceptance and one Order per Revision; atomic replay returns Order |
| Create file upload intent | Required | One active intent/object key for key and purpose/target |
| Finalize upload | **Required** | One finalized object/version per upload intent |
| Submit payment proof | **Required** | One Payment Submission for clean proof Attachment; proof Attachment unique |
| Verify/reject payment | **Required** | One decision per submission; one verified decision per Order |
| Production transition | **Required** | Order/state/version plus unique Production sequence |
| Confirm fulfilment handoff | **Required** | One completion per Fulfilment; proof identity unique |
| Send Message | **Required** | Conversation/sender/client-message key unique |
| Mark Notification read | Optional | Setting the same read fact is naturally idempotent |
| Publish/hide CMS/catalog content | Required | Resource version/state and publication uniqueness |
| Activate configuration revision | Required | One active revision for key/effective instant |

GET/HEAD operations are safe and do not create business state. A UI retry never repeats an action by issuing a second command key before determining the first outcome.

## 5. Acceptance transaction

Quotation acceptance is the strongest idempotency boundary:

1. Authenticate/authorize the quoted Customer.
2. Resolve the idempotency record and request digest.
3. Lock/version-check Quotation, current Revision, and Submitted Request.
4. Confirm current `SENT` state and absence of a terminal response.
5. Enforce unique Acceptance and Order constraints.
6. Create Acceptance, Order, Order Item/price/terms snapshots, initial Payment/Production/Fulfilment records, Audit Events, and outbox intents in one transaction.
7. Mark idempotency result committed with the resulting Order reference.

Concurrent calls with the same or different keys converge to one Order. Any failure before commit creates none of those business records. A timeout after commit is recovered by repeating the same key or querying the accepted revision/Order.

## 6. Payment and production

- Upload intent/finalization idempotency concerns object receipt; it does not verify payment.
- A clean proof Attachment can produce at most one Payment Submission.
- A Payment Submission can receive at most one current Manager decision under approved policy.
- An Order can have at most one verified Payment Verification.
- A repeated Manager verify request returns the existing result only when actor/command digest matches; conflicting outcome returns a domain conflict.
- Production transition keys are scoped to Order and requested command. The current state and expected version are rechecked in the transaction.
- Production start also rechecks the verified decision; no idempotency record can bypass that gate.

## 7. Optimistic concurrency

Mutable aggregate responses expose an opaque ETag derived from stable ID and `record_version`. Commands that can overwrite a mutable draft or transition a current state require `If-Match` or an equivalent expected version.

- Match: command may proceed and increment version once.
- Mismatch: return `VERSION_CONFLICT` with the authorized current version/ETag hint.
- Missing required precondition: return `VERSION_PRECONDITION_REQUIRED`.

A user who reloads and intentionally changes their request uses a new idempotency key. Reusing the old key with a new version/body is a key-reuse error.

Targeted row/advisory locks supplement optimistic concurrency for acceptance, parent-scoped revision/sequence allocation, Payment Verification, and other short critical sections. Locks are never held across provider/network calls.

## 8. Durable events and notifications

- Each authoritative business transition writes one stable domain event/outbox dedupe key in the same transaction.
- Notification creation is unique by event, recipient, and channel.
- In-app and email delivery are separate idempotent jobs so one channel can retry independently.
- A retry records another delivery attempt, not another Notification/business event.
- Resend receives a stable provider idempotency key derived from the internal delivery identity. Internal uniqueness remains authoritative beyond any provider key-retention window.
- The business transition is never repeated to resend a notification.

## 9. Job leasing and replay

Jobs are processed at least once:

1. Atomically claim an eligible job with owner token and lease expiry.
2. Recheck handler version and target state.
3. Perform the bounded idempotent effect.
4. Record attempt and complete or schedule retry atomically.
5. If a worker dies, the expired lease is reclaimed.

Handlers use their own effect/delivery uniqueness. A lease prevents concurrent ownership; it does not itself prove the provider effect occurred. Ambiguous provider timeouts reconcile by querying provider status where supported or retrying under the same provider/internal key.

## 10. Provider webhooks

- Verify signature and replay/timestamp requirements before accepting the event.
- Store a digest and unique `(provider, provider_event_id)` when present.
- Duplicate events return success after confirming prior processing, preventing unnecessary provider retry.
- Out-of-order events use a monotonic provider state mapping or event occurrence time without regressing an already stronger terminal state.
- A different provider event ID carrying the same semantic effect is harmless because business/delivery uniqueness is also enforced.
- Webhooks never directly accept a quotation, verify a bank transfer, start production, approve content, or bypass local authorization.

## 11. File events

Upload finalization binds exactly one expected S3 object key/version, byte size, detected type, and checksum to the intent. Repeated finalization of the identical version returns the existing File Object. A different version/metadata under the same intent is rejected.

GuardDuty clean/malicious/failed results are deduplicated but appended to scan event history where they represent distinct provider observations. A duplicate clean event cannot create a second Attachment or Payment Submission. An older or unknown result cannot regress `CLEAN` or promote a quarantined file without an approved recovery action.

## 12. Idempotency storage and retention

The idempotency record stores no full sensitive request/response. It contains scoped key, request digest, operation, target, status, safe HTTP outcome, resulting resource references, timestamps, and correlation.

Retention duration remains an operational/configuration decision. It must satisfy all of:

- not shorter than the supported client/network retry window;
- business uniqueness remains permanent where the fact must be unique;
- expired records cannot permit duplicate commercial outcomes;
- privacy and storage policies are observed.

Acceptance/order, immutable payment decisions, Production Updates, and fulfilment completion rely on permanent business constraints, not the continued existence of an idempotency cache row.

## 13. Failure cases

| Failure point | Authoritative result |
|---|---|
| Before transaction | No business state; retry same key |
| Transaction rolls back | No partial business state/outbox; retry same key after correction/transient recovery |
| Commit succeeds, response lost | Retry same key returns committed result |
| Provider timeout after request | Internal delivery remains pending/ambiguous; reconcile/retry same effect key |
| Job dies after provider success | Retry/reconcile under same internal/provider key; no duplicate business intent |
| Idempotency store unavailable because DB unavailable | Fail closed; do not execute mutation without durability |
| Stale expected version | No change; reload and obtain explicit user intent before new key |

## 14. Verification

Concurrency tests issue simultaneous same-key and different-key commands, inject failures before/after each transaction stage, lose responses after commit, duplicate/reorder provider events, expire job leases, and verify exact counts of business records, Audit Events, outbox events, Notifications, and provider effects.
